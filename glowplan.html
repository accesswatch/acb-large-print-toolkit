GLOW Accessibility Platform Plan

Purpose

This document defines a definitive, implementation-ready plan to evolve GLOW Site Audit into a high-confidence accessibility platform that combines:

  1. Fast in-app scanning for day-to-day use.
  2. Deep rendered scanning for modern JavaScript-heavy pages.
  3. Workflow-native issue and remediation loops for teams using GitHub and AI-assisted development.
  4. A scalable user experience that avoids tab overload and supports long-term growth.

This plan does not change the current UI immediately. It defines the next architecture and product phases.

Strategic Goals

  1. Improve detection quality without sacrificing scan speed.
  2. Reduce cognitive load for first-time users while preserving advanced controls.
  3. Support users at every level: beginner, practitioner, accessibility specialist, manager, and enterprise platform owner.
  4. Create a stable foundation for future AI-driven workflows when those features are enabled.
  5. Keep accessibility as the core mission for all features.

Outcome Targets (12-Month)

The following targets describe expected outcomes if this plan is executed in full:

  1. 30 to 50 percent increase in high-confidence findings on JavaScript-heavy pages.
  2. 25 to 40 percent reduction in time from finding creation to actionable remediation ticket.
  3. 20 to 35 percent reduction in repeated regressions for routes under automated monitoring.
  4. 40 to 60 percent reduction in “where do I start” friction for first-time users (measured by guided flow completion rate).
  5. 2x increase in recurring scan adoption through scheduled monitoring and easier run presets.

External Inputs Incorporated

This plan takes the strongest ideas from:

  1. Accessibility Insights for Web: fast checks plus guided assessment for manual verification.
  2. Jamal Mazrui’s urlCheck: rendered browser scans, rich evidence, criterion rollups, and report discipline.
  3. GitHub Accessibility Scanner: plugin-based scan model, issue lifecycle workflow, and optional Copilot assignment.
  4. Pa11y CI: sitemap-based scanning, threshold controls, reporter patterns, and URL transform options.
  5. Lighthouse CI: profile-driven assertions, URL-pattern rules, and historical trend reporting.
  6. Microsoft axe pipeline samples: SARIF-friendly CI outputs and practical automation patterns.

Guiding Product Principles

  1. Accessibility-first by design: every new workflow must improve accessibility outcomes, not just add controls.
  2. Progressive disclosure: simple defaults for most users, advanced tuning only when needed.
  3. Evidence over noise: each finding should include enough context to fix quickly.
  4. Human-in-the-loop always: automation suggests and prioritizes; humans validate and approve.
  5. Repeatable and auditable: every run must be reproducible, explainable, and comparable over time.

User Levels And Benefits

The following list summarizes user groups and immediate value.

  1. New users: guided goals, presets, and clear next actions.
  2. Practitioners: faster triage with richer evidence and filtering.
  3. Accessibility specialists: stronger control over rules, criteria mapping, and exceptions.
  4. Team leads and managers: trend dashboards, SLA views, and grouped issue visibility.
  5. Enterprise platform owners: policy controls, governance hooks, CI integration, and secure run handling.

UX Modernization Plan (Tabs To Task-Oriented Accordion Experience)

Problem

The current tab model is growing and can become too wide, visually dense, and cognitively daunting. As more capabilities are added, tabs increase discovery friction and overwhelm.

Design Objective

Replace horizontal tab sprawl with a task-oriented information architecture that:

  1. Prioritizes user intent over product internals.
  2. Uses progressive disclosure for complexity.
  3. Scales to AI-era features without horizontal crowding.

Proposed Structure (No Immediate Code Change)

1) Mission Hub Landing

A single home entry with prominent action cards:

  1. Audit a Site
  2. Fix and Validate
  3. Monitor and Govern
  4. Export and Share
  5. AI Assist (when enabled)

2) Accordion-Based Work Areas

Each mission opens a vertical accordion with collapsible sections.

  1. Discover Scope
  2. Choose Scan Depth
  3. Run and Observe
  4. Review Findings
  5. Route to Action

Each accordion section can be expanded independently and supports inline help text, recommended defaults, and advanced controls hidden by default.

3) Sticky Run Context Rail

A right-side context rail (or lower dock on narrow viewports) shows:

  1. Current run state
  2. Scope summary
  3. Selected profile and overrides
  4. Quick actions (cancel, download, open issue sync)

4) Progressive Advanced Drawer

An “Advanced” drawer per mission contains low-frequency controls (auth context, plugin arrays, assertion matrices, CI export policy).

5) Role-Adaptive Presets

First-run users see starter presets; experienced users can switch to “Power Mode” to expose denser controls and saved templates.

Why Accordion And Task Hub Wins

  1. Vertical space scales better than horizontal tab width.
  2. Users focus on one goal at a time.
  3. Complex controls stay available without overwhelming default views.
  4. Mobile and narrow screens remain usable with less horizontal compression.
  5. Future AI panes can be added as contextual modules instead of new top-level tabs.

AI-Era UX Expansion Strategy (Feature-Flagged)

The following capabilities are designed for future activation when AI features are turned on.

  1. Inline triage assistant in findings panel:
    • Summarize finding severity and likely fix path.
    • Explain why rule fired in plain language.
  2. “Draft fix” action with explicit confidence and risk labels:
    • High confidence: small HTML/ARIA fixes.
    • Medium confidence: style and structural suggestions.
    • Low confidence: manual-review-required advisories.
  3. Batch campaign planner:
    • Group by rule family, route, or ownership.
    • Generate prioritized remediation queue.
  4. Human approval checkpoints:
    • No automatic merge behavior.
    • Clear reviewer sign-off states.
  5. AI observability:
    • Track suggestion acceptance rate.
    • Track rollback rate and reasons.
    • Use outcomes to tune prompt/instruction quality.

Inputs, Outputs, And Operational Contracts

Canonical Run Inputs

  1. Scope inputs: URLs, sitemap, include/exclude patterns, depth, subdomain rules.
  2. Engine inputs: fast/rendered/hybrid mode, viewport, motion, color scheme.
  3. Rule inputs: profile, include/exclude rules, tags, severity gate, plugins.
  4. Auth inputs: login config or auth context.
  5. Workflow inputs: issue sync options, grouped issues, CI export settings.

Canonical Run Outputs

  1. Summary JSON with deterministic schema.
  2. Findings CSV and JSON with criterion mapping.
  3. Evidence artifacts (optional): screenshots, HTML snapshots, node images.
  4. Session log with URL classification and reason codes.
  5. SARIF export (optional) for CI and code scanning tools.
  6. Governance artifacts (optional): issue sync result map and lifecycle deltas.

Canonical Outcomes

  1. Immediate: actionable prioritized findings with reproducible evidence.
  2. Mid-term: stable remediation throughput and reduced duplicate effort.
  3. Long-term: fewer regressions, measurable trend improvement, and clear compliance posture.

Risks And Mitigations

  1. Risk: Feature creep increases complexity.
    • Mitigation: task hub IA, profile-first defaults, advanced drawer containment.
  2. Risk: False positives reduce trust.
    • Mitigation: rule tuning profiles, dedupe signatures, exception registry with expiry.
  3. Risk: Slow scans reduce adoption.
    • Mitigation: default fast mode, hybrid selective deep scan, persistent background jobs.
  4. Risk: AI suggestions applied unsafely.
    • Mitigation: confidence labels, mandatory human review, no auto-merge policy.
  5. Risk: Governance integration adds setup burden.
    • Mitigation: optional integration paths and guided setup checklists.

Implementation Readiness Checklist

  1. Confirm normalized findings schema.
  2. Define profile catalog and default mappings.
  3. Define plugin sandbox and execution guardrails.
  4. Define durable job model and cleanup policy.
  5. Define issue lifecycle synchronization semantics.
  6. Define SARIF and trend data contracts.
  7. Define AI feature flags and review controls.
  8. Validate proposed IA with rapid user walkthrough tests.

Final Recommendation

Execute this plan in waves, beginning with profile-first simplification and engine upgrades, then governance integration, then AI acceleration.

This path balances immediate UX relief, stronger scan quality, and long-term extensibility. It addresses feature creep now through information architecture, while creating a clear runway for future AI-enabled experiences without overwhelming users.
